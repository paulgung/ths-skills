# Code Comments · 代码注释指南

帮助团队编写真正有价值的代码注释：解释**为什么**，而不是"是什么"。
涵盖注释哲学、五种注释层级的写法、多语言示例，以及现有注释的质量审计流程。

```bash
# 只安装此 skill
npx skills add paulgung/ths-skills --skill "code-comments"

# 安装整个 ths-skills 仓库
npx skills add paulgung/ths-skills
```

## 触发方式

在 Cursor 对话中使用以下语句即可激活：

- `帮我给这段代码加注释`
- `审查一下这个文件的注释质量`
- `我们项目的注释规范应该怎么写`
- `这个函数的注释怎么写比较好`
- `帮我写 JSDoc / docstring`
- `把项目里糟糕的注释找出来`

**触发关键词**：写注释、加注释、注释规范、注释审查、comment、docstring、JSDoc

## 核心理念

> **注释的唯一价值是传达代码本身无法传达的信息。**

| ✅ 应该写的注释 | ❌ 不应该写的注释 |
|---------------|----------------|
| 解释为什么这样设计 | 把代码翻译成自然语言 |
| 记录已踩过的坑 | `// i 加 1` |
| 说明非直觉的行为 | 作者名/修改日期（git 的职责） |
| 架构权衡取舍 | 注释掉的大段死代码 |
| 有上下文的 TODO | `// TODO: fix`（没有追踪） |

## 覆盖的注释层级

| 层级 | 适用位置 | 核心内容 |
|------|----------|----------|
| **文件头** | 每个源文件顶部 | 职责描述、设计决策、模块关系 |
| **函数/方法** | 公共 API、复杂函数 | 契约、副作用、边界情况 |
| **内联** | 非直觉的单行代码 | 为什么这样写、魔法数字含义 |
| **架构决策** | 重要设计选择处 | 方案对比、权衡取舍 |
| **TODO/FIXME** | 技术债务标记 | 原因 + 负责人 + issue 链接 |

## 支持的任务模式

### 模式 1：新增注释

告诉 Agent 你想给哪段代码加注释，它会按层级逐一补全：

```
帮我给 userAuthContext.tsx 这个文件加完整的注释，包括文件头和所有公共函数
```

### 模式 2：注释质量审计

对现有代码进行六维度评分（每项 0-10 分），输出结构化审计报告：

```
审查一下 src/api/ 目录下所有文件的注释质量，给出得分和改进建议
```

六个审计维度：

| 维度 | 说明 |
|------|------|
| Why not What | 解释原因而非复述代码 |
| 密度合理 | 注释/代码行比 15-20% |
| 无禁止内容 | 无作者/日期/注释掉的死代码 |
| 文档质量 | 函数注释与签名一致 |
| 时效性 | 注释与代码行为匹配 |
| 清理遗留 | 无无意义 TODO、无 deprecated 残留 |

### 模式 3：制定注释规范

为团队输出一份 `COMMENTS_GUIDE.md` 注释规范文档：

```
帮我们团队制定一份 TypeScript + Python 项目的注释规范
```

## 使用示例

**示例 1：审查注释质量**

```
用户: 帮我审查 src/payment/ 目录下的注释质量

Agent:
注释质量审计报告
================
总分：6.2 / 10

维度得分：
  Why not What   : 5/10  [发现 8 处废话注释]
  密度合理       : 7/10  [实际 23%，略高于目标 20%]
  无禁止内容     : 4/10  [发现 3 处注释掉的死代码]
  ...

主要问题：
  🔴 CRITICAL  payment.ts:47  注释掉的代码块（应通过 git 管理）
  🟠 HIGH      stripe.ts:82   注释与代码行为不符（已修改逻辑未更新注释）
  ...
```

**示例 2：写函数注释**

```
用户: 帮我给这个函数写注释
  function retry(fn, times, delay) { ... }

Agent:
/**
 * 对异步函数执行指数退避重试。
 *
 * 第 n 次重试等待 delay * 2^(n-1) 毫秒，避免在服务压力时雪崩。
 * 所有重试耗尽后抛出最后一次的错误。
 *
 * @param fn    - 需要重试的异步函数
 * @param times - 最大重试次数（不含首次调用）
 * @param delay - 首次重试基础等待时间（毫秒）
 * @returns 首次成功调用的返回值
 * @throws  最后一次失败的错误
 */
```
